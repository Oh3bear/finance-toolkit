## 一、内部往来核对

### 1.1 背景与用途

**场景**：建筑企业集团内部，各分子公司（利润中心）之间存在大量内部交易。比如 A 公司向 B 公司提供服务，A 记账一笔收入，B 记账一笔成本。月底对账时，需要逐笔核对两边的发生额是否一致。

**解决什么问题**：
- 手工核对效率低——一个利润中心对可能有上百笔交易
- SAP 导出的明细账中，内部客商用编码表示，人工识别容易遗漏
- 传统做法是 Excel VLOOKUP + 透视表，但复杂的一对多、多对多匹配很难自动化

**核心能力**：自动将两方（甚至多方）的明细账分录按利润中心对分组，并通过 **M:N 金额匹配算法** 找到借贷双方的对应关系，输出对符/未对符明细。

### 1.2 使用流程

整个流程分为四个步骤：**导入 → 映射 → 核对 → 结果**。

#### 步骤一：导入数据

上传两个 Excel 文件：

| 文件 | 必填 | 说明 |
|------|------|------|
| 明细账 | 是 | SAP 导出的总账明细，需包含：公司代码、凭证编号、行项目、过帐日期、利润中心、科目号、借方/贷方本位币金额、客户/供应商名称及编码 |
| 映射表 | 否 | 可选。也可以在步骤二中手动维护 |

> **注意**：SAP 导出的明细账中，贷方金额已为负数，系统会自动计算净发生额 = 借方 + 贷方。

#### 步骤二：维护映射

这一步定义"哪些科目、哪些客商"参与核对。详见 [1.3 映射表详解](#13-映射表详解)。

核心操作：

1. **科目映射**：勾选参与核对的科目（通常是内部往来的借贷科目）
2. **客商映射**：将客户/供应商名称映射到对应的利润中心编码和名称
3. **智能推断**（推荐）：点击"智能推断内部客商"按钮，系统自动从利润中心名称匹配客商

#### 步骤三：执行核对

确认参数无误后，点击"开始核对"。系统会逐组执行匹配算法，进度条展示当前进度。

#### 步骤四：查看结果

核对结果以三个视图呈现：

| Tab | 内容 |
|-----|------|
| 按利润中心汇总 | 每个利润中心与各对方单位的往来对账全貌 |
| 对符明细 | 已匹配的组，可展开查看每笔对应关系 |
| 未对符明细 | 未完全匹配的组，列出已匹配和未匹配的行 |

顶部统计卡片显示总组数、对符/未对符组数、差异金额。支持导出含三个 Sheet 的 Excel 文件。

### 1.3 映射表详解

#### 科目映射

| 字段 | 说明 |
|------|------|
| 科目编码 | SAP 中的科目号，如 `60010101` |
| 科目名称 | 科目中文名称，如 `内部往来-收入` |

**用途**：只有在此表中配置的科目才会参与核对。用于过滤掉非内部往来科目。

#### 客商映射

| 字段 | 说明 |
|------|------|
| 客商名称 | SAP 明细账中的客户/供应商名称 |
| 利润中心编码 | 该客商对应的利润中心编码，如 `8710003` |
| 利润中心名称 | 利润中心的中文名称 |
| 标准化名称 | （可选）标准化的显示名称 |

**用途**：搭建"客商 → 利润中心"的对应关系桥。因为 SAP 明细账中：
- 甲方公司的分录里，客商字段是乙方名称
- 乙方公司的分录里，客商字段是甲方名称
- 利润中心字段始终是本公司

所以必须有这个映射表，才能将两方的分录归入同一个"利润中心对"中。

#### 智能推断逻辑

点击"智能推断内部客商"后，系统按以下规则自动匹配：

| 优先级 | 规则 | 示例 |
|--------|------|------|
| L1 精确匹配 | 客商名称 = 利润中心名称 | 利润中心名"中建香港" → 匹配客商"中建香港" |
| L2 去本部匹配 | 利润中心名去"-本部"后缀后匹配 | 利润中心名"中建香港-本部" → 匹配客商"中建香港" |
| 备选手动核实 | 编码为 7 位的未匹配客商（内部客商通常为 7 位编码） | 列出供人工勾选确认 |

**注意**：
- 不进行简繁转换——简体和繁体视为不同名称
- 不使用模糊/包含匹配，仅精确匹配和去"-本部"匹配

### 1.4 核对结果解读

#### 组状态

| 状态 | 含义 |
|------|------|
| 对符 | 该组内所有分录均已找到对应关系，净额合计为零 |
| 未对符 | 存在无法匹配的分录，或净额合计不为零 |

#### 匹配类型

| 类型 | 含义 | 典型场景 |
|------|------|----------|
| 汇总零值 | 组内借方合计 = 贷方合计（整体平衡），但未做逐笔拆分 | 快速对符路径 |
| 1:1 | 一笔借方精确对应一笔贷方 | 最常见的情况 |
| 1:N | 一笔对应多笔（如一张发票分多次付款） | 分次结算 |
| M:N | 多笔对应多笔（如多张发票合并付款） | 批量结算 |

### 1.5 基本原理

#### 数据清洗 (cleanData)

1. 统一客商名称：优先取客户名称，无则取供应商名称
2. 计算净额：净发生额 = 借方 + 贷方（SAP 导出中贷方已为负数）
3. 判定方向：净额 > 0 为借方，< 0 为贷方

#### 映射过滤 (filterByMappings)

仅保留：
- 科目号在科目映射集合中
- 客商名称在客商映射集合中

的行。

#### 分组 (groupByProfitCenterPair)

对每条分录：
1. 通过客商映射找到对方利润中心
2. 按本方 + 对方利润中心编码排序后拼接为分组键

这样 A→B 和 B→A 的分录归入同一组。

#### M:N 匹配算法

采用**两轮匹配**策略：

**第一轮：1:1 精确匹配**

构建"金额 → 行索引"的映射表，遍历查找绝对值相等、符号相反的配对。

**第二轮：子集零和匹配**

使用带剪枝的 DFS 回溯算法，在未匹配的行中寻找净额之和为零的子集。

剪枝优化：
1. 当前累计和与候选值同号且绝对值更大 → 跳过
2. 剩余未匹配行的绝对值之和小于当前差值 → 剪枝
3. 按绝对值降序排列，优先匹配大额（减少搜索空间）

算法找到子集后标记为已匹配，迭代执行直到无新子集。

**快速路径**：如果组内所有净额之和约等于零（< 0.01），直接整体标记为"汇总零值"对符，跳过逐笔匹配。

## 二、批量提取 Excel

### 2.1 用途

从大量格式相同的 Excel 文件中提取特定单元格的值（如合同金额、日期、项目编号等）。

### 2.2 使用流程

1. **上传样本**：选择一个代表性的 Excel 文件
2. **配置规则**：用以下两种方式定位目标单元格：
   - **偏移定位**：点击一个锚点单元格，设行/列偏移量
   - **行列搜索**：输入行关键词和列关键词，系统自动定位交叉点
3. **添加多条规则**：每个文件可能需要提取多个字段
4. **批量上传**：选择所有待处理的 Excel 文件
5. **开始处理 & 导出**

### 2.3 基本原理

- 使用 SheetJS 将 Excel 解析为二维数组
- 以样本文件的定位规则为依据，在每个批量文件的相同位置提取数据
- 提取结果汇总为表格，可导出为 Excel

## 三、数据清洗

### 3.1 用途

批量清洗 Excel 数据中的格式问题，适用于建筑行业常见的客商名称不统一、金额格式混乱等场景。

### 3.2 支持的清洗操作

| 操作 | 说明 | 示例 |
|------|------|------|
| 去除首尾空格 | 去掉单元格前后的空格 | `" ABC "` → `"ABC"` |
| 转大写 | 全部字母转大写 | `"abc"` → `"ABC"` |
| 转小写 | 全部字母转小写 | `"ABC"` → `"abc"` |
| 首字母大写 | 每个单词首字母大写 | `"abc def"` → `"Abc Def"` |
| 提取数字 | 提取第一个数字（含小数、正负号） | `"¥1,234.56"` → `"1234.56"` |
| 去除符号 | 去货币符号、逗号、空格 | `"¥ 12,345"` → `"12345"` |
| 查找替换 | 支持普通文本和正则表达式 | `"ABC-001"` → `"ABC_001"` |

### 3.3 使用流程

1. **上传样本**：选择任意 Excel 文件查看数据预览
2. **选择列**：点击列头选中要清洗的列
3. **添加规则**：选择清洗操作，点击"添加规则"
4. **预览效果**：点击"预览清洗效果"查看前后对比
5. **批量处理**：上传多个文件，点击"开始清洗"
6. 处理完成后每个文件自动下载（文件名加 `_清洗后` 后缀，原文件不变）

### 3.4 注意事项

- 同一列可添加多条规则，按添加顺序依次执行
- 查找替换支持正则表达式（勾选"使用正则"）
- 原始文件不会被修改，清洗结果另存为新文件

## 四、PDF 合并

### 4.1 用途

将多个 PDF 文件合并为一个，支持按文件名自动分组、插入分隔页、添加页码。

### 4.2 使用流程

1. **上传 PDF**：拖拽或选择多个 PDF 文件
2. **排序**：可按文件名排序或手动拖拽调整顺序
3. **分组（可选）**：按文件名规律将文件分成多组，每组生成一个独立 PDF

   | 分组方式 | 说明 |
   |----------|------|
   | 按分隔符 | 按文件名中的 `_`、`-` 等分隔符切割，选其中一段作为分组键 |
   | 按固定长度 | 从文件名指定位置截取固定长度 |
   | 按样例/正则 | 输入一个文件名样例，系统自动推断正则；或手动输入 |

4. **选项**：插入分隔页、添加页码
5. **下载**：可下载全部或选中组

### 4.3 分组原理

以样例模式为例：输入 `871-05BMK`，系统自动推断正则 `(\d+)-\d+[A-Z]+`，其中第一个捕获组 `(\d+)` 作为分组键。所有匹配该键的文件归入同一组。

应用正则时采用"**先提取，再分组**"的策略——每份文件先用正则提取分组键，键相同的自动归入同一组，不会出现跨组混入的情况。

## 五、PDF 2 合 1

### 5.1 用途

将 PDF 的每两页合并为一页——可用于将 A4 对开文件转为横版阅读，或创建并排对比视图。

### 5.2 使用流程

1. **上传 PDF**：选择多个 PDF 文件
2. **选择布局**：
   - **左右并排**（side-by-side）：适合宽屏阅读
   - **上下堆叠**（stacked）：适合长文档
3. **点击处理**：每个文件独立生成输出文件（文件名加 `_2to1` 后缀）

### 5.3 注意事项

- 奇数页的文件最后一页独占半页（另一半留空）
- 原文件不会被修改
- 每个输入 PDF 独立输出一个结果文件

## 附录：常见问题

### Q1: 为什么结果"全对符"但感觉不对？

通常是因为**净额计算有误**。检查 Excel 中贷方金额是否已为负数（SAP 导出格式），以及"净发生"列是否存在。

### Q2: 智能推断包括了外部客商怎么办？

设计上已经做了防护：只从利润中心名称出发去匹配客商，外部客商因为利润中心和客商名称不匹配而被排除。7 位编码的未匹配客商被归入"备选手动核实"。

### Q3: 数据会上传到服务器吗？

不会。所有处理均在浏览器中完成，数据不出本机。页面底部有"纯客户端处理"标识。

### Q4: 支持哪些 Excel 格式？

支持 `.xlsx` 和 `.xls` 格式。
